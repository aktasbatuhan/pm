# Multi-tenant Schema Design (Postgres + RLS)

## Scope and source of truth

This document defines the Postgres multi-tenant target schema for Dash and the migration plan from current local SQLite stores.

SQLite schema inventory was derived from current database initialization code in:

- `workspace_context.py` (`workspace.db`)
- `tools/pm_brief_tools.py` (`workspace.db`)
- `tools/pm_goal_tools.py` (`integrations.db`)
- `tools/pm_kpi_tools.py` (`integrations.db`)
- `server.py` (`integrations.db`, plus PM/reporting/auth/integration tables)
- `kai_state.py` (`state.db`, session history)

No implementation details here; design only.

## 1) Tenant model

### 1.1 Identifier shape

Use **both**:

- `tenants.id UUID PRIMARY KEY` (internal relational key; immutable)
- `tenants.slug CITEXT UNIQUE NOT NULL` (human-readable stable handle, used in URLs/admin UX)

Rationale:

- UUID is efficient/safe for joins and avoids accidental guessability.
- Slug is operator-friendly and useful for support/debug workflows.

### 1.2 Generation

- `id`: generated by Postgres (`gen_random_uuid()`).
- `slug`: generated at tenant creation from workspace/org name, lowercased + normalized; collision-safe with numeric suffixing.
- `created_by`: user UUID from Supabase Auth.

### 1.3 Tenant Ōåö user relationship

- Many-to-many via `tenant_memberships`.
- A user can belong to multiple tenants.
- A tenant can have multiple users.
- Tenant-scoped role attached to each membership row.

### 1.4 Core tenant tables

```sql
create table tenants (
  id uuid primary key default gen_random_uuid(),
  slug citext not null unique,
  name text not null,
  status text not null default 'active',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  created_by uuid references auth.users(id)
);

create table tenant_memberships (
  tenant_id uuid not null references tenants(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  role text not null check (role in ('owner','admin','member','viewer','service')),
  is_default boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  primary key (tenant_id, user_id)
);
```

## 2) User model (Supabase Auth + membership + roles)

### 2.1 Auth source

- Authentication is handled by **Supabase Auth** (`auth.users`).
- App data references `auth.users(id)`.

### 2.2 App profile

```sql
create table user_profiles (
  user_id uuid primary key references auth.users(id) on delete cascade,
  display_name text,
  avatar_url text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### 2.3 Role semantics

- `owner`: billing/security/admin authority for tenant
- `admin`: operational admin (can manage integrations, settings)
- `member`: read/write day-to-day operations
- `viewer`: read-only
- `service`: non-human automation identity

### 2.4 Effective tenant context

- All tenant-scoped queries must include `tenant_id` in row filter via RLS.
- Client/session chooses active tenant; server verifies membership for `auth.uid()`.

## 4) RLS policy shape

### 4.1 Standard pattern

1. Resolve `auth.uid()` from Supabase JWT.
2. Verify membership in `tenant_memberships`.
3. Filter rows where `row.tenant_id` is in callerŌĆÖs memberships.

### 4.2 Reusable helper

```sql
create or replace function app.is_tenant_member(target_tenant uuid)
returns boolean
language sql
stable
security definer
set search_path = public
as $$
  select exists (
    select 1
    from tenant_memberships tm
    where tm.tenant_id = target_tenant
      and tm.user_id = auth.uid()
  );
$$;
```

### 4.3 Worked example RLS policy (SQL)

```sql
alter table briefs enable row level security;

create policy briefs_select
on briefs
for select
using (app.is_tenant_member(tenant_id));

create policy briefs_insert
on briefs
for insert
with check (app.is_tenant_member(tenant_id));

create policy briefs_update
on briefs
for update
using (app.is_tenant_member(tenant_id))
with check (app.is_tenant_member(tenant_id));

create policy briefs_delete
on briefs
for delete
using (app.is_tenant_member(tenant_id));
```

Apply the same pattern to all tenant-scoped tables.

## 5) Migration order (cold before hot, with parallelization)

### Phase 0 ŌĆö Foundations (must happen first)

- Create `tenants`, `tenant_memberships`, `user_profiles`.
- Create RLS helper function and role grants.
- Create tenant-scoped skeleton tables with `tenant_id` + indexes.

### Phase 1 ŌĆö Cold/low-churn tables

Migrate first (minimal write contention):

- `workspace_blueprint`, `workspace_learnings`, `workspace_meta`
- `integrations` metadata rows, `github_installations`
- `report_templates`, historical `reports`
- `changelogs`, `team_pulse`, `signal_sources`

### Phase 2 ŌĆö Medium-churn tables

- `goals`, `goal_snapshots`
- `kpis`, `kpi_values`, `kpi_flags`
- `signals`

### Phase 3 ŌĆö Hot-path tables (last)

- `briefs`, `brief_actions`
- `workspace_threads`
- `scheduled_tasks` / `scheduled_task_runs`
- `agent_sessions`, `agent_messages`, `agent_memory_entries`

### Parallelizable work

Can run in parallel once foundations are ready:

- (A) Workspace-context lane: blueprint/learnings/threads/meta
- (B) PM analytics lane: goals + KPI + signals + reports
- (C) Identity/integrations lane: membership, integration connections, credentials

Hot-path backfills should use dual-write period + validation queries before cutover.

## 6) Integration credentials: storage/access decision + threat model

### 6.1 Decision

Use a **split-secret model**:

- Postgres table `integration_connections` stores tenant/platform metadata and **vault reference IDs only**.
- Actual secrets (GitHub installation tokens/private keys, Linear API keys, PostHog keys) live in **Supabase Vault / managed KMS-backed secret store**.
- App retrieves secrets at runtime via privileged server-side path (never exposed to client JWT).

### 6.2 Why this over encrypted text columns

- Encrypted columns still require key management; if key sits with app env, blast radius remains broad.
- Vault/KMS separation provides auditable access and tighter key custody.

### 6.3 Threat model

#### If Supabase/Postgres is compromised

- Attacker may obtain tenant metadata + vault reference IDs.
- Without vault/KMS decryption path and service credentials, raw secrets are not directly recoverable.
- Mandatory mitigations: short-lived tokens where possible, secret rotation playbook, anomaly monitoring.

#### If Railway/app runtime is compromised

- Attacker may abuse app service role to request decrypted secrets (runtime exfil risk).
- Mitigations: least-privilege service roles, outbound allowlists, short token TTLs, aggressive rotation, runtime audit logs.

#### Comparative impact

- Supabase compromise risk is primarily **data exposure + metadata intelligence**.
- Railway compromise risk is primarily **active secret misuse/exfiltration**.
- Therefore: optimize for minimizing runtime credential lifetime and enforce rapid revocation.

## Open questions (requires operator decision)

1. **Tenant slug mutability:** immutable forever vs admin-editable with redirect/audit log?
2. **Cross-tenant operators:** do internal support users need scoped break-glass access, and how is it audited?
3. **Service identities:** one global service user with per-tenant grants vs one service principal per tenant?
4. **Cron execution model:** keep external scheduler as source-of-truth or fully move schedule state into Postgres tables?
5. **Session backfill tenancy:** authoritative mapping for legacy `state.db` sessions with no explicit workspace/tenant key.
6. **Credential backend finalization:** Supabase Vault availability/limits in target plan tier vs external KMS fallback.
7. **Data retention policy:** per-table retention windows (messages, signals, snapshots) and tenant-configurable overrides.
8. **PII boundaries:** whether message/session content is allowed in centralized Postgres for all tenants by policy/compliance.
